Chronos-Graph

A bitemporal-native graph database purpose-built for RAG and AI-agent memory. Every fact tracks valid time and transaction time, contradictions are invalidated (not deleted), and retrieval returns LLM-ready, cited context.

面向 RAG 与 AI-agent memory 打造的 bitemporal-native 图数据库。每条 fact 同时记录 valid time 与 transaction time,矛盾事实被 invalidate 而非删除,检索直接返回带 citation、可供 LLM 使用的上下文。

v0.1.0 Rust REST · MCP · Embedded Apache-2.0

What this engine does without an LLM: there is no built-in embedding model, so the SIMILAR(...) operator ranks facts with a real BM25 index over their verbalized text. Everything else — bitemporal storage, atomic upsert, point-in-time queries, provenance/citations, communities, entity resolution, multi-tenancy and durability — is implemented and tested.

没有 LLM 时它能做什么: 引擎不内置 embedding model,因此 SIMILAR(...) 用真实的 BM25 index 对 fact 文本排序。其余能力——bitemporal storage、atomic upsert、point-in-time query、provenance/citation、community、entity resolution、multi-tenancy 与 durability——均已实现并测试。

What it does & what it solves它能做什么、解决什么

Chronos-Graph is the memory layer for RAG pipelines and AI agents. You write facts as they are learned; it remembers when each fact was true and when you learned it, keeps contradictions instead of overwriting them, and answers queries with cited, point-in-time, LLM-ready context — in one call.

Chronos-Graph 是 RAG 流水线与 AI agent 的 memory layer。你随学随写 fact,它会记住每条 fact 何时为真、何时被获知,矛盾事实只保留不覆盖,并在一次调用里返回带 citation、point-in-time 正确、可直接喂给 LLM 的上下文。

TEMPORAL

Time-travel memory时间旅行式记忆

Store facts with both valid time and transaction time, then ask what you knew as of any past instant.

为 fact 同时记录 valid time 与 transaction time,可查询任意历史时刻「当时已知什么」。

NON-LOSSY

History is never lost历史永不丢失

Contradictions invalidate the old fact instead of deleting it, so the full timeline stays queryable.

矛盾事实只 invalidate 旧 fact 而非删除,完整时间线始终可查。

PROVENANCE

Cited answers可引用的答案

Every retrieved line traces back to its source doc/chunk — no unsourced claims.

每条检索结果都可回溯到来源 doc/chunk,不凭空捏造。

HYBRID

One-call retrieval一次调用完成检索

BM25 full-text, graph traversal and temporal filtering combine in a single extended-Cypher query.

BM25 全文、graph 遍历与 temporal 过滤在一条 extended-Cypher query 中合并完成。

LLM-READY

Context, not rows输出上下文而非数据行

Token-budgeted subgraph selection serializes to graph-to-text the model can read directly.

按 token budget 选子图并 graph-to-text 序列化,直接产出模型可读的上下文。

AGENT-NATIVE

Multiple front doors多种接入方式

Use it over REST, as a built-in MCP server, or as an embedded Rust library — same kernel, with tenant isolation and durable RocksDB.

可经 REST、内置 MCP server 或嵌入式 Rust library 使用——同一 kernel,带 tenant 隔离与持久化 RocksDB。

When to use it适用场景

  • Long-term memory for LLM agents — remember, update and time-scope what they learn, without losing the audit trail.
  • LLM agent 的长期 memory——记住、更新并按时间界定所学,同时保留可审计的历史。
  • GraphRAG with citations — grounded answers where every claim links to a source and contradictions are handled explicitly.
  • 带 citation 的 GraphRAG——每个论断都连到来源、矛盾被显式处理的可溯源问答。
  • Temporal knowledge base / audit log — query the state of the world "as of" any past instant for compliance or debugging.
  • 时态知识库 / 审计日志——为合规或排查,查询任意历史时刻「当时」的世界状态。
  • Multi-tenant SaaS memory — one engine, hard isolation per tenant, durable on RocksDB.
  • 多租户 SaaS memory——一个 engine、按 tenant 硬隔离、基于 RocksDB 持久化。

  • Fact — 一条 subject predicate object edge,带 bitemporal span 与 provenance reference(doc、chunk)。

Architecture架构

Chronos-Graph pushes temporal logic, hybrid retrieval and provenance down into the engine rather than bolting them onto a general-purpose graph store. The system is a layered kernel: thin interfaces on top, a single engine kernel in the middle, and a pluggable storage/foundation layer underneath.

Chronos-Graph 把 temporal logic、hybrid retrieval 与 provenance 下推进 engine,而非在通用 graph store 之上拼装。整体是一个分层的 kernel:顶层是轻薄的 interface,中间是单一的 engine kernel,底层是可插拔的 storage / foundation。

System overview整体架构纵览

Top-down dependency, bottom-up data flow. Every interface speaks to the same kernel; the kernel never depends on a specific storage backend (it holds a Box<dyn StorageEngine>), so the in-memory MVCC engine and the durable RocksDB engine are interchangeable.

自上而下是依赖方向,自下而上是数据流。所有 interface 都对接同一个 kernel;kernel 不依赖具体 storage backend(持有 Box<dyn StorageEngine>),因此 in-memory MVCC engine 与持久化 RocksDB engine 可互换。

Retrieval pipeline检索流水线

One /v1/search call runs end-to-end inside the engine: the extended-Cypher string is lexed, parsed and planned, then the executors fan out to the BM25 index (for SIMILAR), the bitemporal as_of filter (for AS OF) and the token budgeter (for TRAVERSE SEMANTIC), and the result is serialized to LLM-ready text with citations.

一次 /v1/search 调用在 engine 内端到端完成:extended-Cypher 字符串经 lex / parse / plan 后,executor 分发到 BM25 index(处理 SIMILAR)、bitemporal as_of filter(处理 AS OF)与 token budgeter(处理 TRAVERSE SEMANTIC),最终序列化为带 citation、可供 LLM 使用的文本。

Crate & module mapCrate / 模块依赖图

Edges point from a crate to what it depends on (top depends on bottom). Service crates are thin wrappers over chronos-embedded, which assembles the kernel; chronos-query drives the foundation libraries; and every crate ultimately builds on chronos-common. The same kernel ships as a server, an MCP tool provider, or an in-process library.

箭头从一个 crate 指向其依赖(上层依赖下层)。service crate 只是 chronos-embedded 的轻封装,由它组装 kernel;chronos-query 驱动底层 foundation 库;所有 crate 最终都构建在 chronos-common 之上。同一个 kernel 可作为 server、MCP tool provider 或进程内 library 交付。

Query language查询语言

An OpenCypher subset extended with RAG / temporal operators. It has two surfaces: a read language you send to /v1/search, and write operations expressed through the atomic UPSERT_FACT operation (not the text language). "Delete" and "update" map onto bitemporal operations, not destructive edits.

在 OpenCypher subset 基础上扩展了 RAG / temporal operator。它分两个面:发给 /v1/search 的读语言,以及通过原子 UPSERT_FACT operation 表达的写操作(不走文本语言)。「删除」「更新」都映射为 bitemporal 操作,而非破坏性改写。

Read queries — grammar读查询 — 语法

Every clause is optional and clauses are order-sensitive (top to bottom). Today WHERE only interprets SIMILAR(...); any other predicate is parsed but ignored.

所有 clause 均可选,且顺序敏感(自上而下)。目前 WHERE 只解释 SIMILAR(...);其他谓词会被解析但忽略。

MATCH ( <var> )
[ WHERE SIMILAR(<var>, "text") [ AND SIMILAR(...) ... ] ]
[ AS OF ( VALID | TRANSACTION ) TIME <ms> ]
[ TRAVERSE SEMANTIC( depth <= <n> , budget = <n> tokens ) ]
RETURN ( CONTEXT( cite = true | false )  |  <ident> [ , <ident> ... ] )

Write operations (create · update · delete)写操作(增 · 改 · 删)

Writes go through one atomic operation — UPSERT_FACT — exposed as POST /v1/memory (REST), add_memory (MCP) and FactStore::ingest / upsert_fact (embedded). Within a single engine transaction it detects contradictions, closes invalidated spans, and appends the new fact. The policy selects the conflict behaviour.

写操作只走一个原子 operation —— UPSERT_FACT,对外是 POST /v1/memory(REST)、add_memory(MCP)与 FactStore::ingest / upsert_fact(embedded)。它在单个 engine transaction 内检测矛盾、关闭被 invalidate 的区间、追加新 fact。policy 决定冲突行为。

Aggregation & advanced predicates — not yet聚合与高级谓词 —— 暂未支持

To set expectations clearly, the text language does not yet support the following. They are on the roadmap; for now do them client-side or via the typed embedded API.

为避免误解,文本语言目前尚不支持下列能力。它们在 roadmap 上;暂时请在客户端处理,或用 typed embedded API。

  • General WHERE comparisons / property filters (=, <, >, ranges) — only SIMILAR is interpreted today (planned)
  • 通用 WHERE 比较 / 属性过滤(=、<、>、区间)—— 目前只解释 SIMILAR (planned)
  • Aggregation: COUNT / SUM / AVG / GROUP BY (planned)
  • 聚合:COUNT / SUM / AVG / GROUP BY (planned)
  • ORDER BY / LIMIT / SKIP (planned)
  • ORDER BY / LIMIT / SKIP (planned)
  • Variable-length / multi-hop path patterns ((a)-[*1..3]->(b)) and relationship-typed patterns (planned)
  • 变长 / multi-hop path pattern((a)-[*1..3]->(b))与带关系类型的 pattern (planned)
  • CREATE / SET / DELETE / MERGE statements — writes go through UPSERT_FACT instead (by design)
  • CREATE / SET / DELETE / MERGE 语句 —— 写入统一走 UPSERT_FACT (设计如此)
  • A vector kNN query operator — the brute-force cosine index exists but isn't wired into the query language yet (planned)
  • 向量 kNN query operator —— brute-force cosine index 已存在,但尚未接入查询语言 (planned)

Multi-tenancyMulti-tenancy

Every fact carries a TenantId. The tenant filter is pushed into the scan, so a tenant-scoped request can never observe another tenant's facts, and contradiction detection never crosses the boundary.

每条 fact 都带 TenantId。tenant filter 被下推进 scan,因此限定 tenant 的请求绝不会看到其他 tenant 的 fact,contradiction detection 也不会跨 tenant。

curl -XPOST localhost:8080/v1/memory \
  -H 'content-type: application/json' -H 'X-Tenant-Id: 42' \
  -d '{"subject":"Alice","predicate":"lives_in","object":"Tokyo","valid_from":1000,"doc":1,"chunk":1}'

curl -XPOST localhost:8080/v1/search \
  -H 'content-type: application/json' -H 'X-Tenant-Id: 42' \
  -d '{"query":"MATCH (n) WHERE SIMILAR(n,\"Alice\") RETURN CONTEXT(cite = true)"}'

Omitting X-Tenant-Id targets the default tenant (0). The same header works on /v1/communities and /v1/resolve.

省略 X-Tenant-Id 即 default tenant(0)。该 header 同样适用于 /v1/communities 与 /v1/resolve。
